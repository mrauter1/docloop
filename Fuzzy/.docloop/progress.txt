- Writer pass: rewrote `SAD.md` from a high-level overview into a normative architecture spec. - Verifier pass 2026-03-17T02:30:29Z: `SAD.md` is coherent and appropriately scoped, but it is not implementation-ready yet because a few implementation-critical contracts remain underdefined.
- The main gap is the adapter boundary in Section 7. Define the canonical normalized request shape and the canonical adapter response/error surface, not just the fact that they exist. At minimum, pin down the required request fields the core passes to adapters, what the adapter returns to the core (`str` raw candidate, structured object, or another exact shape), and the provider-error categories that must remain distinct. Without this, two competent implementations could both claim conformance while one adapter returns raw text and another returns parsed JSON, or one raises transport/auth errors while another returns them as ordinary content failures.
- The schema contract also needs to be tightened. Sections 5.2, 5.4, and 6.4 should state exactly which schema forms are supported and under what runtime requirements: which JSON Schema dialect/subset is in scope, what "supported model type" means, and whether `dispatch.input_schema` / `output_schema` use the same schema/model contract as `extract` or are JSON-Schema-only. Without that, one implementation could accept only JSON Schema dicts, while another accepts Pydantic models for command schemas, and both would appear consistent with the current text.
- The retry/configuration surface needs one normative home. Sections 4.4 and 9.2 refer to a maximum/configured attempt count, but the document does not specify where that value comes from, whether callers can override it per call, or whether `1` means "no repair" while `0` is invalid or also allowed. Hard-coding one attempt versus five attempts would produce materially different behavior and both are currently plausible. Add a concise configuration contract rather than repeating retry rules per primitive.
- Clarify the `LLMOps` default system prompt feature in Section 8. Either specify whether it is overrideable per call and whether per-call values replace or append to the default, or remove it from the public wrapper contract if it is intentionally internal. As written, implementers would have to guess whether this default is observable API surface or just an implementation convenience.
