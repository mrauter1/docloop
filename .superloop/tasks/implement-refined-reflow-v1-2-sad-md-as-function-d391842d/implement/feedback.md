# Implement ↔ Code Reviewer Feedback

- `IMP-001` (`blocking`): [reflow_runtime/providers.py](/workspace/CodexTest/docloop/reflow_runtime/providers.py#L60) executes shell steps with `["/bin/sh", "-c", "<cmd>"]` while recording and documenting `["/bin/sh", "-lc", "<cmd>"]`. Concrete failure scenario: a workflow that relies on login-shell startup behavior or any environment initialization that `/bin/sh -lc` performs will behave differently at runtime than the persisted `command_argv`, so shell steps can fail in production even though the artifacts claim the normative contract was honored. Minimal fix direction: centralize shell argv construction in one helper and use that exact argv both for `meta.json.command_argv` and for `subprocess.run(...)`.
- `IMP-002` (`blocking`): [reflow_runtime/controller.py](/workspace/CodexTest/docloop/reflow_runtime/controller.py#L288) persists `pending_input` while the run status is still `running`, and later increments `pending_input.auto_round` under the same invalid status in [_run_full_auto_answers](/workspace/CodexTest/docloop/reflow_runtime/controller.py#L435). Concrete failure scenario: if the controller is interrupted between those saves and the later transition to `awaiting_input`, `run.json` violates the SAD invariant that `pending_input` must only exist when `status` is `awaiting_input`, and external `status`/`reply`/recovery logic can no longer trust the persisted state. Minimal fix direction: switch the run to `awaiting_input` before saving any newly created or updated `pending_input`, keep `active.json` in sync while waiting, and only move back to `running` after answers are resolved and the step is about to restart.
- `IMP-003` (`blocking`): [reflow_runtime/controller.py](/workspace/CodexTest/docloop/reflow_runtime/controller.py#L157) turns every `StepFailedError` into an immediate terminal `step_failed`, so malformed control output never gets the required retry path. Concrete failure scenario: an agent emits one malformed `<questions>` block or invalid tagged decision on loop 1 of 3; the SAD requires that iteration to be finalized as failed, the next request to carry a short warning, and the step to retry until the loop budget is exhausted, but the current code aborts the whole run on the first protocol slip. Minimal fix direction: handle protocol-parse failures inside the agent-step path, finalize the iteration, preserve the run as non-terminal while loops remain, and only convert to terminal `max_loops_exceeded` once the retry budget is exhausted.
- `IMP-004` (`blocking`): [reflow.py](/workspace/CodexTest/docloop/reflow.py#L52) and [reflow_runtime/controller.py](/workspace/CodexTest/docloop/reflow_runtime/controller.py#L157) do not catch `KeyboardInterrupt` or install any SIGINT handling, so Ctrl+C bypasses the required stop path entirely. Concrete failure scenario: the operator interrupts a live run, the process exits immediately, and the workspace is left with a stale `active.json` plus a `running` run instead of finalizing the reserved iteration as `interrupted` and terminalizing the run as `stopped` per Section 17.4. Minimal fix direction: route SIGINT/`KeyboardInterrupt` through the same stop/finalization path as `reflow stop`, including reserved-iteration reconciliation and lock cleanup.
- `IMP-005` (`blocking`): [reflow_runtime/controller.py](/workspace/CodexTest/docloop/reflow_runtime/controller.py#L83) promises stop semantics for an active subprocess, but [reflow_runtime/providers.py](/workspace/CodexTest/docloop/reflow_runtime/providers.py#L23) and [reflow_runtime/providers.py](/workspace/CodexTest/docloop/reflow_runtime/providers.py#L68) invoke provider and shell children via blocking `subprocess.run(...)` without ever recording a child PID in `active.json`. Concrete failure scenario: `reflow stop <run_id>` during a long-running provider or shell step sends `SIGTERM` only to the controller process, marks the run `stopped`, and can leave the orphaned child process running and still mutating the workspace after terminalization. Minimal fix direction: centralize child-process launching around `subprocess.Popen`, write the live child PID into `active.json` before waiting, and clear or overwrite it once the child exits so `stop` and Ctrl+C can terminate the actual worker process before marking the run stopped.
- `IMP-006` (`blocking`): [reflow_runtime/protocol.py](/workspace/CodexTest/docloop/reflow_runtime/protocol.py#L43) renders the agent footer with `run.step_loops[step.name]` even though the controller does not reserve the next iteration until afterwards in [_execute_agent_step](/workspace/CodexTest/docloop/reflow_runtime/controller.py#L177). Concrete failure scenario: the first request for a step advertises `- loop: 0` even though the persisted iteration directory is `001`, and every subsequent retry or post-reply restart reports the previous loop number instead of the current one; agents that branch on loop count or write diagnostics against the request footer see inconsistent state, violating the required footer contract. Minimal fix direction: reserve the iteration before rendering `request.txt`, or pass the soon-to-be-reserved loop number explicitly into the request renderer so the footer and persisted iteration metadata stay in sync.
- `IMP-007` (`blocking`): [reflow_runtime/controller.py](/workspace/CodexTest/docloop/reflow_runtime/controller.py#L69) calls `_resolve_pending_input(...)` before entering [_drive_run](/workspace/CodexTest/docloop/reflow_runtime/controller.py#L80), and [reflow.py](/workspace/CodexTest/docloop/reflow.py#L69) still handles the resulting `KeyboardInterrupt` by only printing `"Interrupted."` and returning `27`. Concrete failure scenario: start a run that is `awaiting_input`, invoke `reflow reply <run_id>`, and press Ctrl+C at the inline prompt (or during full-auto answer generation). The CLI exits with code `27`, but the run remains persisted as `awaiting_input` instead of terminalizing as `stopped`; in-process callers also retain `active.json` because no stop/finalization path runs. Minimal fix direction: wrap `reply_to_run`'s pre-`_drive_run` input-resolution phase in the same stop handler used inside `_drive_run`, or register a higher-level SIGINT/`KeyboardInterrupt` guard around any period after `write_active(...)` where the run should be reconciled and terminalized before returning `27`.
- `IMP-008` (`blocking`): [reflow_runtime/policy.py](/workspace/CodexTest/docloop/reflow_runtime/policy.py#L42) treats any repo entry that resolves outside the workspace as a policy violation even when that path did not change in the iteration. Concrete failure scenario: a repository already contains a benign symlink like `escape -> /shared/cache`; running a no-op shell step (`cmd: true`) still fails terminally with `step_failed` and `failure_reason: "escape: resolves outside workspace"`, even though SAD Section 9.1 only requires a violation when a changed path escapes the workspace. Minimal fix direction: centralize escape detection in the same changed-path diff used for policy evaluation so unchanged pre-existing symlinks do not fail steps, while writes through escaping paths are still surfaced as violations.
- `IMP-009` (`blocking`): [reflow_runtime/policy.py](/workspace/CodexTest/docloop/reflow_runtime/policy.py#L13) prunes `.reflow/` entirely from `snapshot_workspace()`, so child-process writes to controller-owned state are invisible to policy enforcement. Concrete failure scenario: a buggy or malicious `shell`/`agent` step edits `.reflow/config.yaml`, `.reflow/active.json`, or run artifacts; the before/after snapshots stay identical, `allow_write` / `forbid_write` checks never fire, and the run can continue with tampered controller state. Minimal fix direction: centralize Reflow-owned path filtering so only controller-authored writes are exempted, while child-step mutations under `.reflow/` are still diffed and rejected as `step_failed`.
